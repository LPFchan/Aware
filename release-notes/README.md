# Release Notes

Files in this folder provide the canonical release notes for tagged releases.

## Naming

- Create one Markdown file per release tag for **English**: the filename must match the tag exactly, for example `v1.5.md` for tag `v1.5`.
- Optionally add localized notes for the same tag (same pattern: `v1.5.<lang>.md`):
  - Korean: `v1.5.ko.md`
  - Japanese: `v1.5.ja.md`
  - Chinese (Simplified): `v1.5.zh-Hans.md`  
  If these files are missing, Sparkle still publishes English-only release notes HTML for those languages.

## How It Is Used

- The release workflow checks for `release-notes/<tag>.md` when publishing a tag.
- If the file exists, GitHub Releases uses it as the release body instead of autogenerated notes.
- When Sparkle appcast publishing is enabled, the workflow also:
  - Converts `release-notes/<tag>.md` to `release-notes/<tag>.en.html` on GitHub Pages.
  - Converts optional `release-notes/<tag>.ko.md`, `.ja.md`, and `.zh-Hans.md` to matching `.html` files when present.
  - Patches `appcast.xml` with `sparkle:releaseNotesLink` elements using `xml:lang` (`en`, plus any of `ko`, `ja`, `zh-Hans` that were published) so the Sparkle updater shows the language that matches the user’s macOS preferred languages.

If `release-notes/<tag>.md` is missing at publish time, English HTML falls back to the GitHub Release body (same as before).

## Recommended Structure

Use this shape unless a release needs something different:

```md
## New in X.Y

- User-facing change
- User-facing change

## Notes

- Upgrade or distribution note
- Anything users should know before updating

**Full Changelog**: https://github.com/LPFchan/Aware/compare/vX.Y...vX.Z
```

## Reissuing The Same Public Version

If you need to republish the same public version, such as `1.4`:

1. Keep `MARKETING_VERSION` unchanged.
2. Increment `CURRENT_PROJECT_VERSION` so Sparkle sees a newer build.
3. Update the matching release notes file for that tag.
4. Recreate the GitHub release tag so the workflow republishes the release and appcast.

The appcast generation script removes any existing item with the same short version before inserting the reissued update, so the feed keeps a single entry for that public version.
